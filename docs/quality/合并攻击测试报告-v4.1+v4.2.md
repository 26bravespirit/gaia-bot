## Round 1：v4.1 × v4.2 交叉干扰测试

### 🔴 CR-01：S5 执行顺序冲突 — Anti-AI 改写 vs Memory Blur 时序矛盾

**攻击向量：** v4.1 定义 S5 执行流为"Anti-AI 规则链 → 现有 S5 流程"。v4.2 定义 memory_blur 位置为"在 S5 或 S4.5 中执行"。合并后谁先谁后？

**具体冲突：**
- blur 先于 Anti-AI → R04 长度截断把模糊化表达切掉，效果归零
- blur 后于 Anti-AI → R05 已把传记叙述压缩成一句，blur 找不到时间标记（正则匹配失败）

**严重度：** 🔴 P0

**修复建议：** 明确 S5 内部执行链为 `Anti-AI R01-R06 → memory_blur → 现有 S5 流程(口头禅/错别字/emoji/拆分)`。

---

### 🔴 CR-02：R04 长度截断制造"用户不可见事实"

**攻击向量：** `max_message_length: 60`。Bot 生成 150 字传记叙述 → S4.5 从原始输出提取 3 个事实（正确，S4.5 在 S5 前执行）→ R04 截断到 60 字 → 用户只看到前 2 个事实 → DB 存了全部 3 个 → 下次 S2 检索到第 3 个事实并注入 Prompt → bot 引用了用户从未听过的信息。

**严重度：** 🔴 P0

**修复建议：** S4.5 提取事实时增加 `user_visible` 标记。方案 A：S4.5 在 S5 之后再标记哪些事实被截断（需要 S5 反馈截断位置）。方案 B：S2 检索时区分"已向用户展示的事实（高置信引用）"和"生成但被截断的事实（低置信引用）"。

---

### 🟠 CR-03：R05 知识讲解压缩器误判传记叙述

**攻击向量：** R05 检测"包含 >3 个逗号分隔长句 + 不含情绪词 → 知识讲解体"。传记叙述"高一的时候转学到了香港，刚开始语言不太适应，后来慢慢习惯了，那时候开始接触冲浪"完全符合 R05 匹配模式。

**结果：** 传记叙述被压缩成一句话，丢失多个可提取事实。

**严重度：** 🟠 P1

**修复建议：** R05 增加豁免条件：如果 S2 的 ContextBundle 标记 `biography_topic: true`，则 R05 降低阈值或跳过。

---

### 🟠 CR-04：Anti-AI + 传记双重 Prompt 注入的 Token 预算竞争

**攻击向量：** 合并后 System Prompt 约 1250 tokens（persona_summary 300 + Anti-AI 约束 200 + 传记事实 200 + forbidden + blur 指令 50 + 其余上下文 500）。指令过多导致 LLM 注意力稀释，可能忽略部分约束。

**严重度：** 🟠 P1

**修复建议：** 定义 Prompt 预算上限 1500 tokens，各模块硬配额。Anti-AI 高 strictness 模式下传记事实从 8 条降为 5 条。

---

### 🟠 CR-05：S5.5 BLOCK 重写循环触发 double-blur

**攻击向量：** ai_score ≥ 60 → BLOCK → 回退 S5 最高强度重写 → S5 重新执行包含 memory_blur → 双重模糊化（"好像是...好像是..."）。

**严重度：** 🟠 P1

**修复建议：** S5.5 BLOCK 重写时设置 `blur_already_applied: true` 标记，跳过二次 blur。

---

### 🟡 CR-06：`incomplete_answer` vs 传记一致性

**攻击向量：** 用户问多个传记问题，0.3 概率触发 incomplete_answer → bot 只答部分 → S4.5 从原始输出提取全部事实 → DB 存的比用户看到的多。

**严重度：** 🟡 P2（同 CR-02，但触发频率较低）

---

### 🟡 CR-07：S5.5 knowledge_density 对传记内容的误报

**攻击向量：** 传记回复必须包含具体时间、地点、事件（信息密度高）→ knowledge_density 贡献 ai_score +10。

**严重度：** 🟡 P2

**修复建议：** S5.5 增加上下文感知：`biography_topic: true` 时 knowledge_density 权重从 10 降为 5。

---

## Round 2：合并后 Pipeline 逻辑一致性校验

合并后完整 Pipeline: `S1 → S2(enhanced) → S3+S4 → [S4.5] → S5(enhanced) → [S5.5] → S6`

### 🔴 PL-01：S3+S4 Prompt 注入顺序未定义

**攻击：** S3+S4 需同时注入 4 块内容：基础人设、Parameter Interpreter 结果、Anti-AI 约束、传记约束。注入顺序影响 LLM 遵从度（recency bias：靠后的内容优先级更高）。当前无任何文档定义此顺序。

**严重度：** 🔴 P0

**修复建议：** 定义 `prompt_assembly_order`：`persona_summary → biography_constraints → anti_ai_constraints → parameter_interpreter_fragments`。传记约束靠前（保证一致性），Anti-AI 约束靠后（利用 recency bias）。

---

### 🔴 PL-02：S5 内部三功能执行链未定义

**攻击：** S5 现承担三项任务（Anti-AI 规则链、memory_blur、原有口头禅/错别字/拆分），输入输出类型不一致（R01-R06 操作 string，现有 S5 输出 StyledMessages[]，memory_blur 需要 BiographicalFact[] 参数），需要内部 sub-pipeline 定义。

**严重度：** 🔴 P0（与 CR-01 关联，此处强调接口类型不一致问题）

**修复建议：** 定义 S5 内部执行链及类型转换点：
```
S5.step1: Anti-AI Rules (string → string)
S5.step2: Memory Blur (string + BiographicalFact[] → string)
S5.step3: 口头禅/错别字/填充词 (string → string)
S5.step4: 消息拆分 + emoji/sticker (string → StyledMessages[])
```

---

### 🟠 PL-03：S4.5 异步写入 vs S2 同步读取的竞态条件

**攻击场景：** 用户快速连发两条消息（间隔 <3 秒），消息 A 的 S4.5 异步写入可能还未完成，消息 B 的 S2 就开始检索 → 检索不到 A 刚生成的传记事实。

**严重度：** 🟠 P1

**修复建议：** S4.5 采用 write-through 策略：异步写入 DB 同时同步更新 in-memory recent_facts 缓存，S2 检索先查缓存。

---

### 🟠 PL-04：降级路径未覆盖新增 Stage

**攻击：** 降级路径（S3+S4 超时 → S5 万能模板 → S6）中，万能模板进入增强版 S5 会被 Anti-AI 规则链误改写，且经过 S5.5 评分无意义。

**严重度：** 🟠 P1

**修复建议：** 降级路径显式跳过 Anti-AI、blur、S5.5：`降级: S1 → S2 → [S3 超时] → S5(仅 step3+step4) → S6`。

---

### 🟡 PL-05：S3+S4 合并 PART A/B 结构缺少传记决策字段

**攻击：** `biography_relevant: boolean` 应属于 PART A（认知决策），当前未定义。

**严重度：** 🟡 P2

---

### 🟡 PL-06：事件总线并发消费竞态

**攻击：** S5 的 `anti_ai.rule_triggered` 和 S4.5 的 `biography.fact_extracted` 可能同时触发记忆层状态更新。

**严重度：** 🟡 P2

**修复建议：** 事件消费者使用 FIFO 队列串行化处理。

## Round 3：完整性攻击（缺失组件检测）

### 🔴 CP-01：`human_behaviors` 概率行为无执行机制

**缺失：** v4.1-branch 定义了 5 个概率行为（incomplete_answer 0.3, topic_drift 0.1, selective_ignore 0.2, push_back 0.15, mood_refusal 0.05），但未定义在哪个 Stage 执行、以何种方式执行。

**严重度：** 🔴 P0

**修复建议：** 定义为 S3/S4 Prompt 层行为指令，由 Parameter Interpreter 按概率决定本次是否注入对应指令。理由：topic_drift 和 selective_ignore 必须在认知层面决定。

---

### 🔴 CP-02：memory_blur 的 `blur_triggers` 未实现

**缺失：** 定义了 `blur_triggers: [specific_date, exact_sequence, low_importance_detail]`，但 `applyMemoryBlur()` 实现只做随机概率模糊 `Math.random() > config.blur_rate`，未使用 blur_triggers。所有传记引用均 15% 概率模糊化，无论话题类型。

**严重度：** 🔴 P0

**修复建议：** 先判断 trigger 类型 → 命中 trigger 才应用 blur_rate → 未命中不模糊化。

---

### 🟠 CP-03：10+ 工具函数引用但未定义

**缺失：**
- v4.1-branch: `extractFirstPointOnly()`, `truncateToHumanLength()`, `compressToOneLiner()`, `replaceWithNaturalEmpathy()`
- v4.2: `extractKeywords()`, `extractTopicKeywords()`, `groupByPeriod()`, `contradicts()`, `normalizePeriod()`, `stringSimilarity()`, `groupByTopicKeywords()`

**严重度：** 🟠 P1

**修复建议：** v5 增加"辅助函数规范"附录，至少为 `contradicts()` 和 `stringSimilarity()` 提供算法选型建议。

---

### 🟠 CP-04：`applyMemoryBlur()` 参数来源未定义

**缺失：** 函数需要 `relatedFacts: BiographicalFact[]` 参数，但 S5 输入定义（v4 原文）为 `RawReply + TemporalState + RelationshipState`，不包含 BiographicalFact[]。

**严重度：** 🟠 P1

**修复建议：** 扩展 S5 输入为 `RawReply + TemporalState + RelationshipState + BiographicalContext`。

---

### 🟠 CP-05：`biography.writeback.enabled = false` 行为未定义

**缺失：** 三种配置模式下 S4.5、S2 检索、Cron Job 的行为均未显式声明。

**严重度：** 🟠 P1

**修复建议：** 明确三种模式：① biography 整块不存在 → 全部关闭；② anchors 有 + writeback false → S2 检索 anchors，S4.5 不执行；③ writeback true → 完整功能。

---

### 🟡 CP-06：constraints.yaml 缺少传记相关约束

**缺失：** 缺少 anchors.age_range 不重叠、forbidden_fabrications 不与 anchors 矛盾、max_total_facts ≥ anchors 总数、blur_rate + strictness 高值冲突警告。

**严重度：** 🟡 P2

---

### 🟡 CP-07：MemoryDAO 新方法无签名定义

**缺失：** 7 个新方法只有名称，无 TypeScript 方法签名和参数定义。

**严重度：** 🟡 P2

### 🔴 UJ-01：多子问题场景 R01 误判

**场景：** 用户问"你小时候在哪个区？上什么学校？课余做什么？"
**问题：** Bot 回答多项内容被 R01 识别为"AI 式列举"→ `extractFirstPointOnly()` 只保留第一项 → 传记信息大量丢失 + CR-02 同时触发。

**严重度：** 🔴 P0

**修复建议：** R01 增加多问题豁免检测：如果用户消息包含多个问句（≥2 个问号或疑问词），R01 降低 confidence 或跳过。

---

### 🟠 UJ-02：身份试探 + 传记追问组合攻击

**场景：** "你不是 AI 吧？那说说你小时候在哪长大的"
**问题：** identity_check 和传记问题检测无优先级定义。如果两者并行处理 → 回复可能自相矛盾（一边装傻一边给详细信息）。

**严重度：** 🟠 P1

**修复建议：** 定义 `identity_check: triggered` 时 response_strategy 强制为 deflect，传记问题本轮不回答。

---

### 🟡 UJ-03：memory_blur 对 anchor 事实的保护依赖链

**场景：** 用户伪造矛盾"你不是说大一才测 MBTI 吗？" → anchor 明确是高三 → 但如果 blur 触发 → bot 对锚点事实产生动摇。

**问题：** blur 代码中有 `generatedRefs = relatedFacts.filter(f => f.source_type === 'generated')`（正确排除 anchor），但此代码依赖 `relatedFacts` 被正确传入 S5 → 回到 CP-04。

**严重度：** 🟡 P2

---

### 🟡 UJ-04：Anti-AI 约束导致传记内容生成不足

**场景：** 所有传记问题的回答都被 ≤60 字约束限制。长期累积下，S4.5 可提取的事实变少 → biographical_facts 增长缓慢 → writeback 系统实际效用降低。

**严重度：** 🟡 P2（系统级效率问题，不影响正确性）

---

### 🟡 UJ-05：Cron Job 传记维护 vs Anti-AI 事件日志的数据膨胀

**场景：** Anti-AI 每条消息触发 1-3 个事件（rule_triggered, score_calculated），传记每次触发 1-2 个事件 → 事件日志增长速度快 → Cron Job 是否需要清理事件日志？当前未定义。

**严重度：** 🟡 P2

---

### 🟡 UJ-06：S4.5 Haiku 调用失败的降级路径

**场景：** S4.5 使用 Haiku 轻量调用提取事实 → Haiku API 超时或返回空 → 当前设计中 S4.5 是异步不阻塞 → 提取失败静默忽略？还是回退到 rule-based？未定义。

**严重度：** 🟡 P2

**修复建议：** 明确 S4.5 降级链：`Haiku 调用 → [超时 3s] → rule-based 提取 → [失败] → 跳过本次提取 + 发射 biography.extraction_failed 事件`。

## 合并可行性评估

**当前状态：** 🟡 **有条件可合并**

v4.1-branch 和 v4.2 在**设计层面无根本矛盾**，核心问题集中在：
1. **执行顺序未定义**（S5 内部、Prompt 注入顺序）— 设计补充即可

3. **部分功能定义不完整**（human_behaviors 执行、blur_triggers 实现）— 需补全

**建议合并路径：**
1. 先修复全部 7 个 P0 问题 → 产出 v5-draft
2. v5-draft 进行二次攻击测试（聚焦 P0 修复效果 + P1 项）
3. 确认通过后正式发布 v5
