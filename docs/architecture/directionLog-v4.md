## 决策摘要

| 方向 | 选择 | 执行顺序 |
|------|------|---------|
| A3 人设即代码 | ✅ 已选 | 第 1 优先 |
| A2 管道模型重构 | ✅ 已选 | 第 2 优先（A3 完成后） |
| 其余 8 个方向 | ⏸ 存档 | 留待后续版本评估 |

**选择理由：** A3 解决"人设配置散落各层"的结构性问题（同时修复 C3 配置依赖混乱），A2 将概念架构转化为可执行的管道模型（同时修复 C1 LLM 调用爆炸）。先 A3 后 A2 的顺序是因为：管道模型的每个 stage 需要知道"从哪里读配置"，而 Persona Schema 正是这个答案。

**思维模式选择：** A 组（收敛深化）。当前阶段架构已经稳定，需要从"设计正确"转向"设计可执行"。

## A 组 · 第一性原理方向（5 个）

### [A1] 最小可运行人格（Minimum Viable Persona）

**状态：** ⏸ 存档

**核心洞察：** 砍掉所有非必需层，只保留一对一私聊所需的最小组件集，一周内跑通端到端。

**难度：** ⭐⭐ | **突破潜力：** 🔥🔥🔥

**存档原因：** 与 A3/A2 不冲突。当 A3+A2 完成后，可以基于 Persona Schema + Pipeline 快速提取 MVP 子集。实际上 A3+A2 完成后做 A1 会更高效——因为你有了声明式配置和可插拔管道，"砍功能"就是"去掉几个 stage + 简化 persona.yaml"。

**潜在回归时机：** v4 完成后，作为 v5 的落地验证方向。

---

### [A2] 数据流重构：管道模型（Pipeline-First Redesign）

**状态：** ✅ 已选（第 2 优先）

**核心洞察：** 把架构从"概念分层"重构为"执行管道"，每个 stage 是可插拔处理器，明确输入/输出 schema，区分 LLM-required vs rule-based stage。

**难度：** ⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥

**选择理由：** 直接将 v3.1 的概念架构转化为可编码的执行模型。解决 C1（LLM 调用合并）。与 A3 形成互补——A3 定义"配置从哪来"，A2 定义"消息怎么流"。

**执行依赖：** 依赖 A3 的 Persona Schema 作为各 stage 的配置输入。

---

### [A3] 人设即代码（Persona-as-Code）

**状态：** ✅ 已选（第 1 优先）

**核心洞察：** 把人设从散落各层的隐式配置抽象为统一的声明式 Schema。换人设 = 换配置文件。系统从"特定角色 bot"变为"人格引擎"。

**难度：** ⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥

**选择理由：** 解决 C3（配置依赖混乱），同时为 A2 的管道模型提供配置基座。是 v4 架构升级的基础。

**执行依赖：** 无前置依赖。

---

### [A4] 记忆是核心产品（Memory-Centric Redesign）

**状态：** ⏸ 存档

**核心洞察：** 把记忆层从"支撑层"提升为"核心层"，深入设计采集、压缩、检索、遗忘、联想 5 个子系统。

**难度：** ⭐⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥🔥

**存档原因：** 这是突破潜力最高的方向之一，但需要先有稳定的架构骨架（A3+A2）才能深入。在没有 Persona Schema 的情况下设计记忆系统，会面临"记什么取决于人设"的耦合问题。

**潜在回归时机：** v5 或 v6，作为体验深度突破的核心方向。

---

### [A5] 终局倒推：1000 天后的关系（End-State Backcast）

**状态：** ⏸ 存档

**核心洞察：** 从 1000 天后的运行状态倒推，设计遗忘策略、高原期应对、人设微成长、重大事件响应。

**难度：** ⭐⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥

**存档原因：** 长期设计在系统尚未运行时无法验证。建议在系统实际运行 1-3 个月后再回到这个方向——那时会有真实数据支撑设计决策。

**潜在回归时机：** 系统上线运行后，基于实际数据驱动的优化方向。

---

## B 组 · 创意驱动方向（5 个）

### [B1] 跨界移植：内部家庭系统（IFS Transplant）

**状态：** ⏸ 存档

**核心洞察：** 认知层拆分为 2-3 个"子人格"（共情者/理性者/守护者），通过内部协商产生决策，自然产生犹豫和矛盾。

**难度：** ⭐⭐⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥🔥

**存档原因：** 极具创造性但实现复杂度最高。需要先有稳定的单一认知层管道（A2），再考虑拆分为多子人格。

**潜在回归时机：** v6+，作为拟人度突破的终极方向。

---

### [B2] 极端假设：情绪自主性（Emotional Autonomy Hypothesis）

**状态：** ⏸ 存档

**核心洞察：** 给 bot 内生情绪波动（3-7 天周期），低谷期表现更安静、高峰期更活跃。

**难度：** ⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥

**存档原因：** 实现不难但效果高度依赖参数调校，需要在真实用户交互中验证。适合在系统运行后作为"拟人度微调"来引入。

**潜在回归时机：** 系统上线后，时间引擎的参数优化阶段。可在 Persona Schema 中预留 `emotional_cycle` 配置项。

---

### [B3] 视角翻转：红队测试协议（Adversarial User Perspective）

**状态：** ⏸ 存档

**核心洞察：** 从"试图识破 bot 的用户"视角，建立 20 个红队测试场景和自动化测试框架。

**难度：** ⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥

**存档原因：** 需要先有可运行的系统才能做红队测试。建议在 MVP 上线后立即启动。

**潜在回归时机：** MVP 上线后的第一个优化周期。

---

### [B4] 反向构建：可控不完美矩阵（Inversion Design）

**状态：** ⏸ 存档

**核心洞察：** 列出"最快破功的7种方式"，反转为设计约束——系统必须主动制造可控的不完美。

**难度：** ⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥🔥

**存档原因：** 可以部分融入 A3 的 Persona Schema（不完美参数矩阵作为人设配置的一部分）。完整实现留待后续。

**潜在回归时机：** A3 设计时预留接口，v5 详细实现。

---

### [B5] 组合创新：共同经历生成器（Shared Experience Engine）

**状态：** ⏸ 存档

**核心洞察：** 工具调用结果 → 分享 → 用户反馈 → 沉淀为"共同经历"长期记忆，制造关系粘性。

**难度：** ⭐⭐⭐⭐ | **突破潜力：** 🔥🔥🔥🔥🔥

**存档原因：** 依赖记忆系统（A4）和工具调用的成熟度。是 A4 的自然延伸。

**潜在回归时机：** A4 完成后，作为记忆系统的高级应用。

## v4.1-branch · Anti-AI-Speech 防御机制

**状态：** ✅ 已设计
**触发来源：** MVP 实测核心发现 — 对话口气明显像 AI
**决策日期：** 2026-04-04

### 问题描述
MVP 测试中发现关键体验问题：机器人说话口气特别像 AI，表现为：
1. 回复中提供多种选择供对方参考
2. 末尾提出选择疑问句（"你觉得呢？""需要展开吗？"）
3. 单条回复过长，像知识讲解
4. 开场白模板化（"好的""当然"）

### 解决方案
三层防御架构：
- **第一层（S3/S4 Prompt 注入）：** persona.yaml 新增 `anti_ai_speech` 配置，Parameter Interpreter 生成 Anti-AI 行为约束指令注入 System Prompt
- **第二层（S5 结构改写）：** 6 条 Anti-AI 规则链（R01-R06），在现有 S5 流程前执行
- **第三层（S5.5 后置校验）：** 新增 Stage，8 维 AI 指纹评分器，ai_score ≥ 60 触发降级

### 关联文档
- 完整设计：`本体聊天机器人-架构设计-v4.1-Anti-AI-Speech.md`

### 合并条件
- Anti-AI 规则链 false positive 率 < 5%
- ai_score 评分器准确率 > 80%
- S5.5 BLOCK 触发率 < 10%

## v4.2 · 传记记忆与生成一致性机制（Biographical Memory + Consistency Gate）

**状态：** ✅ 已设计
**触发来源：** MVP 实测 — LLM 即兴生成传记细节无返写，窗口滑过后自相矛盾（"高三测MBTI" vs "大一才第一次测"）
**决策日期：** 2026-04-04
**设计者：** GG CUI（原始四层方案）+ AI Coach（盲区补充 ×3）

### 问题递进链
1. 即兴生成无约束 → 每次可能不同
2. 无返写机制 → 说过就忘，窗口外不可追溯
3. 返写无控制 → 人生轨迹膨胀，事实间互相矛盾
4. 无一致性校验 → 新生成内容无法与已有记录对账

### 解决方案（三层 + 容错）
- **第一层：传记锚点（Biographical Anchors）** — persona.yaml 新增 `biography` 字段，作者预定义关键人生节点，注入 System Prompt
- **第二层：生成事实返写（S4.5 新增 Stage）** — 从 LLM 原始输出中异步提取传记事实，写入前冲突检测，上限 50 条
- **第三层：一致性门控（S2 增强）** — 话题关键词检索相关事实，注入 Prompt 约束 LLM 不与已有事实矛盾
- **容错层：记忆模糊化** — `memory_blur_rate` 概率性模糊回忆表达，让偶尔不一致变成"记不清"而非"说谎"

### AI Coach 补充的 3 个盲区
1. **提取时机纠正：** 原方案放在 S5，实际应在 S3+S4 输出后（S5 已改写文本，语义提取准确率低）
2. **生成事实间互相矛盾：** 新增写入前冲突检测（同主题不同时段检测 + 同时段矛盾细节检测）
3. **Cron Job 联动：** 周度传记事实聚类合并 + 时间线矛盾扫描 + 月度密度审计

### 关联文档
- 完整设计：`本体聊天机器人-架构设计-v4.2-Biographical-Memory.md`

### 新增组件
- persona.yaml: `biography` 配置块（anchors, blank_periods, forbidden_fabrications, writeback）
- persona.yaml: `imperfection.memory_blur` 配置块
- SQLite: `biographical_facts` 表
- Pipeline: S4.5 Biographical Fact Extractor（异步，Haiku）
- S2: 传记检索 + Prompt 注入增强
- S5: memory_blur 模糊化逻辑
- 事件总线: 4 个 biography.* 事件
- MemoryDAO: 7 个新方法
