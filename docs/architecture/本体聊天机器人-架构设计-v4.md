## 一、核心设计哲学

### 第一性原理

> **真正让人觉得"像真人"的是时间维度的行为模式，而不是单条回复的质量。**

### v4 新增原则

| 原则 | 含义 |
|------|------|
| **防破功优先** | 一次穿帮归零所有信任 |
| **低频设基调，高频调节奏** | 长期关系决定态度，短期状态调表达 |
| **事件驱动状态更新** | 各层发射事件，记忆层单点写入 |
| **工具是手段不是能力** | 受人设知识边界约束 |
| 🆕 **人设即代码** | 所有拟人行为参数化，配置驱动一切 |
| 🆕 **管道可插拔** | 每个处理阶段独立、可替换、可跳过、可降级 |

## 三、Persona Schema 完整定义

### 3.1 Schema 结构总览

```yaml
persona.yaml
├── version: "1.0"
├── meta                    # 元信息
│   ├── name
│   └── description
├── identity                # 身份与人格
│   ├── background          # 年龄/性别/职业/教育
│   ├── personality_traits  # 大五人格 + 自定义维度
│   └── identity_boundary   # 身份守卫策略
├── knowledge               # 知识边界
│   ├── expertise_domains   # 精通（直接答）
│   ├── familiar_domains    # 熟悉（模糊答，可查）
│   ├── ignorance_domains   # 不懂（承认不会）
│   └── knowledge_style     # 工具调用后的包装方式
├── language                # 语言风格
│   ├── base_style          # 正式度/消息长度/emoji频率
│   ├── vocabulary          # 偏好词/避免词/口头禅
│   └── imperfection        # 错别字率/改正行为/填充词
├── temporal                # 时间行为
│   ├── state_model         # 状态概率模型（替代刚性日程）
│   ├── response_timing     # 各状态下的延迟范围
│   ├── proactive_behavior  # 主动行为配置
│   └── emotional_cycle     # 情绪周期（预留，默认关闭）
├── social                  # 社交行为
│   ├── relationship_stages # 各关系阶段的行为修正参数
│   └── group_chat          # 群聊策略
└── memory                  # 记忆策略
    ├── importance_weights  # 各类记忆的重要度权重
    └── forgetting          # 遗忘策略
```

### 3.3 配置派生链

| Pipeline Stage | 读取的 Schema 字段 | 派生的运行时参数 |
|---------------|-------------------|----------------|
| S1 消息调度 | `temporal.response_timing`, `temporal.schedule_template` | 缓冲窗口 Δt、当前时段状态 |
| S2 上下文组装 | `memory.*`, `identity.background` | 记忆检索权重、自我状态初始值 |
| S3 认知决策 | `identity.identity_boundary`, `knowledge.*`, `social.*` | 身份守卫规则、工具调用阈值、群聊策略 |
| S3.5 工具执行 | `knowledge.knowledge_style.tool_use_phrasing` | 工具结果包装模板 |
| S4 回复生成 | `language.base_style`, `identity.personality_traits` | LLM prompt 中的风格指令 |
| S5 感知包装 | `language.*`, `social.relationship_stages[当前阶段]` | 不完美注入参数、语气修正 |
| S6 出站调度 | `temporal.response_timing`, `language.base_style.avg_message_length` | 消息拆分阈值、间隔时间、打字速度模拟 |

### 3.4 配置管理机制

**校验：** 修改 persona.yaml 时运行 schema validator，检查必填字段、数值范围（0-1）、枚举值合法性

**热加载：** chokidar 监听 persona.yaml → diff 检测变更字段 → 增量通知相关 stage → 无需重启

**版本控制：** persona.yaml 纳入 git，每次修改自动记录 diff，可回溯人设演变历史

## 四、Pipeline 执行模型

### 4.1 Stage 详细定义

#### S1 · 消息调度（Rule-based）

```
输入: 原始消息流 (Lark CLI event)
输出: MessagePackage { messages: [], classification: string, metadata: {} }

逻辑:
  1. 消息进入缓冲区，启动计时器 Δt
     Δt = persona.temporal.response_timing.base_delay_ms[当前时段][0] × 0.3
     (取最小延迟的30%作为缓冲窗口)
  2. Δt 内有新消息 → 重置计时器，追加到缓冲区
  3. Δt 到期 → 分类器运行:
     - urgent_interrupt: 情绪危机关键词 / @提及 / 直接提问 → 跳过缓冲
     - append_type: "还有"/"对了"/纯表情 → 合并到前一条
     - new_topic: 默认分类
  4. 输出 MessagePackage
```

#### S2 · 上下文组装（Rule + DB）

```
输入: MessagePackage
输出: ContextBundle {
  messages,              # 当前消息包
  immediate_memory,      # 本次 session 对话历史
  working_memory,        # 近期对话摘要
  relevant_long_memory,  # 检索到的长期记忆 (top-K)
  relationship_state,    # 当前关系模型快照
  self_state,            # 当前自我状态快照
  temporal_state,        # 时间引擎当前各频率状态
  persona_summary        # 人设摘要（注入 LLM prompt）
}

逻辑:
  1. 从即时记忆读取当前 session 历史
  2. 从工作记忆读取近期摘要
  3. 将消息内容 embed → 在长期记忆中检索 top-K 相关片段
     权重 = persona.memory.importance_weights 加权
  4. 读取关系模型、自我状态、时间引擎状态
  5. 生成 persona_summary（从 persona.yaml 提取关键字段，
     拼接为 LLM 可理解的自然语言描述）
  6. 打包输出 ContextBundle
```

#### S3 · 认知决策（LLM）

```
输入: ContextBundle
输出: DecisionPackage {
  should_reply: boolean,
  emotion_analysis: string,
  response_strategy: string,
  identity_check: "pass" | "triggered",
  identity_strategy: string | null,
  tool_needed: boolean,
  tool_query: string | null,
  topic_update: {} | null,
  events_to_emit: []
}

LLM Prompt 结构:
  [System] 你是{persona_summary}。根据以下上下文做出决策...
  [Context] {ContextBundle 的结构化摘要}
  [Instruction] 分析用户消息，输出 JSON 格式的决策包。

  决策要求:
  - should_reply: 群聊中未被@且话题与你无关时可为 false
  - identity_check: 检查用户是否在试探你的身份
  - tool_needed: 仅当话题在 ignorance_domains 或需要时效信息时为 true
  - 基于 {knowledge.expertise/familiar/ignorance} 决定知识策略

跳过条件:
  if should_reply == false → 跳过 S3.5, S4, S5, S6
  if identity_check == "triggered" → S4 使用特殊生成策略
```

#### S3.5 · 工具执行（API，可选）

```
输入: DecisionPackage.tool_query
输出: ToolResult { raw_result, digest, phrasing }

逻辑:
  1. 根据 tool_query 路由到对应工具（浏览器/搜索API/知识库）
  2. 执行查询，获取原始结果
  3. 摘要提取（可选用小模型）
  4. 从 persona.knowledge.knowledge_style.tool_use_phrasing 随机选取包装语
  5. 输出 ToolResult

跳过条件:
  if DecisionPackage.tool_needed == false → 跳过
```

#### S4 · 回复生成（LLM）

```
输入: DecisionPackage + ToolResult(可选) + ContextBundle
输出: RawReply { content: string, tone: string, self_disclosure_level: number }

LLM Prompt 结构:
  [System] 你是{persona_summary}。当前关系阶段:{relationship_stage}。
           当前情绪状态:{self_state}。
  [Context] 对话历史 + 检索到的记忆
  [Decision] {DecisionPackage 的策略指导}
  [Tool Result] {如有}
  [Instruction] 生成回复。风格要求:
    - 正式度: {language.base_style.formality}
    - 平均长度: {language.base_style.avg_message_length} 字左右
    - 使用偏好词汇: {vocabulary.preferred_words}
    - 避免: {vocabulary.avoided_words}

合并优化:
  当 S3.5 未执行时，S3 和 S4 合并为一次 LLM 调用，
  使用 structured output 同时返回 DecisionPackage + RawReply
```

#### S5 · 感知包装（Rule-based）

```
输入: RawReply + TemporalState + RelationshipState
输出: StyledMessages [{text, has_emoji, has_sticker, sticker_id}]

逻辑:
  1. 语气调整: 应用 relationship_stages[当前阶段].tone_modifier
  2. 口头禅注入: 按 catchphrase_frequency 概率插入
  3. 不完美注入:
     - 按 typo_rate 概率引入错别字
     - 按 incomplete_thought_rate 概率截断句子
     - 如果触发了错别字且 correction_behavior=true，追加一条"*更正"
  4. emoji/sticker 决策: 按频率参数概率附加
  5. 消息拆分: 超过 multi_message_threshold 字则拆分
  6. 输出 StyledMessages 数组
```

#### S6 · 出站调度（Rule-based）

```
输入: StyledMessages + TemporalState(仲裁结果)
输出: ScheduledDelivery [{message, send_at}]

逻辑:
  1. 时间引擎仲裁 → 获取最终延迟参数
  2. 第一条消息: 发送时刻 = now + 仲裁延迟
  3. 先发送"正在输入..."信号 (发送时刻 - typing_simulation_ms)
     typing_simulation_ms = message.length / typing_speed_cpm × 60000
  4. 后续消息: 间隔 = 3000~8000ms (随机)
  5. 输出 ScheduledDelivery，交给基建层定时发送
```

### 4.2 Pipeline 特性

| 特性 | 实现方式 |
|------|---------|
| **可插拔** | 每个 Stage 实现统一接口 `process(input) → output`，可替换 |
| **可跳过** | S3 输出 `should_reply=false` → 跳过后续 stage |
| **可合并** | S3+S4 在不需要工具时合并为一次 LLM 调用 |
| **可降级** | LLM stage 超时 → 跳到预生成回复模板 → S5 → S6 |
| **可观测** | 每个 Stage 的 input/output 自动发射到事件总线 |
| **可扩展** | 新增能力 = 插入新 Stage（如 S3.7 子人格协商） |

### 4.3 降级路径

```
正常路径: S1 → S2 → S3 → [S3.5] → S4 → S5 → S6
LLM 调用合并: S1 → S2 → S3+S4 → S5 → S6          (省 1 次调用)
工具调用: S1 → S2 → S3 → S3.5 → S4 → S5 → S6     (多 1 次 API)
不回复: S1 → S2 → S3(should_reply=false) → END
降级: S1 → S2 → [S3 超时] → S5(万能模板) → S6     (0 次 LLM)
```

## 五、时间引擎（v4 更新）

时间引擎在 Pipeline 模型中的角色：**不是一个 Stage，而是多个 Stage 共享的参数注入器。**

```
时间引擎状态
    │
    ├──→ S1 (消息调度): 提供当前时段 → 影响缓冲窗口 Δt
    ├──→ S2 (上下文组装): 提供时间状态快照 → 写入 ContextBundle
    ├──→ S4 (回复生成): 提供情绪基调 → 影响 LLM prompt
    └──→ S6 (出站调度): 提供仲裁结果 → 决定最终发送时机
```

### 层叠仲裁协议（不变）

```
天-月级(基调) → 小时级(覆盖) → 分钟级(覆盖) → 秒级(中断)
```

### 情绪衰减函数（不变）

```
emotion_intensity(t) = initial_intensity × e^(-λt)
```

## 六、事件总线（v4 更新）

## 七、Parameter Interpreter（v4.1 新增）

### 7.1 参数→Prompt 映射

数值参数不直接传给 LLM，而是通过 `prompt_mappings.yaml` 翻译为自然语言行为描述：

```yaml
# prompt_mappings.yaml 示例
humor_level:
  [0.0, 0.2]: "你很少开玩笑，表达方式严肃直接。"
  [0.2, 0.4]: "你偶尔会幽默一下，但整体风格偏稳重。"
  [0.4, 0.6]: "你有适度的幽默感，对话中自然穿插轻松表达。"
  [0.6, 0.8]: "你很爱开玩笑，经常用调侃、自嘲、夸张的方式表达。"
  [0.8, 1.0]: "你是话题中的搞笑担当，几乎每次回复都带梗。"

formality:
  [0.0, 0.2]: "你说话非常口语化，大量使用网络用语。像发微信语音转文字。"
  [0.2, 0.4]: "你说话随意但通顺，偶尔用网络用语。像和朋友发消息。"
  [0.4, 0.6]: "你说话自然得体，口语和书面语混合。像和不太熟的同事聊天。"
  [0.6, 0.8]: "你说话比较正式，用词准确，句式完整。像和客户对话。"
  [0.8, 1.0]: "你说话非常正式，用词考究，逻辑严谨。像在写邮件。"
```

流程：`persona.yaml` → Parameter Interpreter → `resolved_prompt_fragments{}` → S3/S4 的 system prompt

### 7.2 跨字段约束校验

```yaml
# constraints.yaml 示例
constraints:
  - name: "外向性-主动性一致"
    rule: "extraversion > 0.7 → max_daily_initiations >= 1"
    severity: "warning"
  - name: "正式度-词汇一致"
    rule: "formality > 0.6 → preferred_words 不含网络俚语"
    severity: "error"
  - name: "内向性-群聊一致"
    rule: "extraversion < 0.3 → speak_probability <= 0.2"
    severity: "warning"
```

## 八、S3+S4 合并策略（v4.1 修正）

### 合并条件

```
必须分离调用的场景：
  - 消息含强情感信号（情绪判断必须纯粹）
  - 身份边界可能被触发（安全优先）
  - 需要工具调用（S3→S3.5→S4 三阶段）
  - 群聊需判断"要不要回"
其余场景：可以合并
```

### 合并时的 Prompt 结构约束

```
[Instruction]
严格按顺序完成。先完成 PART A 全部字段后再开始 PART B。

===== PART A: 认知决策 =====
- emotion_analysis, response_strategy, identity_check, topic_update

===== PART B: 回复生成（基于 PART A）=====
- 严格基于 PART A 的决策结果生成回复
```

### 后校验（Rule-based，零 LLM 成本）

合并调用返回后，快速校验决策与生成的一致性。不一致时触发 S4 单独重新生成。

## 九、记忆维护 Cron Job（v4.1 新增）

### 任务清单

| 频率 | 任务 | 内容 |
|------|------|------|
| **每日** | 工作记忆压缩 | >7天的摘要合并精炼；importance>0.8 提升为长期记忆 |
| **每日** | 自我状态衰减 | 清除过期 recent_experiences 和已衰减 active_emotions |
| **每周** | 长期记忆整理 | 聚类合并相似记忆；低重要度+长期未检索→标记"可遗忘"（降权，不删除） |
| **每周** | 承诺追踪审计 | 过期承诺→标记为可主动提及素材；已兑现承诺→归入共同经历 |
| **每月** | 关系模型评估 | 计算30天互动指标 → 关系阶段升降级判断 |
| **每月** | 人设一致性审计 | 抽样20条回复 vs persona.yaml → 一致性评分 + 偏差告警 |
| **每月** | 记忆统计报告 | 各级记忆条目数、检索命中率、可遗忘池大小、趋势对比 |

### 架构位置

```
Pipeline (实时，消息驱动)  ←──→  事件总线  ←──→  记忆层
                                    ↑
Memory Maintenance Cron (离线，时间驱动)
  - 在低活跃时段执行（凌晨 3-5 点）
  - 结果通过 maintenance.* 事件通知，不直接修改实时状态
  - 不干扰实时 Pipeline 运行
```
