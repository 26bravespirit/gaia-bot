## 一、核心设计哲学

### 第一性原理

> **真正让人觉得"像真人"的是时间维度的行为模式，而不是单条回复的质量。**

这意味着：

1. 系统的核心竞争力不在"回复更聪明"，而在"行为更像人"
2. 时间不是一个维度，而是贯穿所有层的轴
3. "不完美"本身是拟人的一部分——精确到不自然就是破功

### 设计原则

| 原则 | 含义 |
|------|------|
| **防破功优先** | 一次穿帮归零所有信任，所以边界守卫的优先级高于体验增强 |
| **低频设基调，高频调节奏** | 长期关系决定"我是什么态度"，短期状态只调"此刻怎么表达" |
| **各层发射事件，记忆层单点写入** | 状态更新通过事件总线异步完成，不破坏分层调用结构 |
| **工具是手段不是能力** | bot 有搜索能力但受人设知识边界约束，不该"什么都知道" |

## 二、系统总架构

```
┌─────────────────────────────────────────────────────────┐
│                     感知表达层                             │
│  拟人输出控制 · 情感语气适配 · 多模态选择                   │
│  (只负责"怎么说"，不决定"说什么")                          │
├─────────────────────────────────────────────────────────┤
│                     认知决策层                             │
│  对话连贯性 · 身份边界守卫 · 场景策略(私聊/群聊)            │
│  工具调用决策 · 自发行为生成器                              │
├─────────────── ⏱ 时间引擎(纵贯轴) ─────────────────────┤
│  天-月级(基调) → 小时级(覆盖) → 分钟级(覆盖) → 秒级(中断)  │
│  层叠仲裁协议 · 情绪衰减函数                               │
├─────────────────────────────────────────────────────────┤
│                   记忆与状态层                              │
│  即时记忆 · 工作记忆 · 长期记忆 · 关系模型 · 自我状态       │
│  自我状态 = 唯一写入者（通过事件总线接收）                   │
├─────────────────────────────────────────────────────────┤
│                    基础设施层                              │
│  消息调度器(缓冲·分类·输出拆分)                             │
│  LLM 调用管线 · 本地持久化(SQLite/LevelDB)                 │
│  工具运行时(浏览器引擎·搜索API·知识库·日历)                 │
│  可观测性 · 安全 · 热更新配置                               │
├─────────────────────────────────────────────────────────┤
│  事件总线 (全局侧通道，异步)                                │
│  各层发射事件 → 记忆层消费并更新状态                         │
│  事件流自动归档 → 可观测性数据源                             │
└─────────────────────────────────────────────────────────┘
```

## 三、各层详细设计

### 3.1 感知表达层

**职责边界：** 只负责"怎么说"，不决定"说什么"。接收认知层的语义决策，输出最终的用户可见消息。

**核心模块：**

| 模块 | 功能 | 输入 | 输出 |
|------|------|------|------|
| 语气适配器 | 根据情绪状态和关系阶段选择表达风格 | 认知层决策 + 时间引擎当前基调 | 风格化文本 |
| 不完美注入器 | 可控地引入错别字、省略号、口语化表达 | 风格化文本 + 人设不完美参数 | 拟人文本 |
| 多模态选择器 | 决定是否附加表情包、图片、语音条等 | 语义内容 + 场景 + 关系亲密度 | 多模态消息组合 |
| 消息拆分器 | 将长回复拆分为多条短消息（已移至消息调度器） | — | — |

**关键设计点：**

- 语气模板不是硬编码的，而是从"人设语料库"中 few-shot 检索
- 不完美注入是概率性的，频率参数可调（默认约 5-8% 的消息有轻微不完美）
- 多模态选择基于场景规则：深夜对话更多用表情包少用文字、群聊用 sticker 回应比文字更自然

---

### 3.2 认知决策层

**职责边界：** 决定"说什么"和"不说什么"。这是系统的"大脑"。

**核心模块：**

#### 3.2.1 对话连贯性引擎

- 维护当前活跃话题栈（多话题可交织）
- 每次回复前检查：是否和之前说过的话矛盾
- 检索相关历史记忆注入 context window

#### 3.2.2 身份边界守卫（断路器）

- **所有回复在发出前必须过此检查**
- 检查项：是否暴露 AI 身份、是否违反人设设定、是否产生不可能的人类行为
- 触发策略：回避 / 幽默转移 / 反问 / 承认模糊（"我不太记得了"）
- 防 prompt injection：识别并忽略试图修改系统行为的输入

#### 3.2.3 场景策略

| 场景 | 策略 |
|------|------|
| 私聊 | 正常对话模式，完整的情感投入 |
| 群聊 | 发言克制模式：判断该不该说、什么时候说、说多少 |
| 被 @ | 跳过群聊克制，但保持群聊简洁风格 |
| 多人同时私聊 | 并行处理，但时间引擎统一调度总回复速度（不能同时秒回所有人） |

#### 3.2.4 工具调用决策

- 接入"人设知识轮廓"：基于人设年龄/职业/兴趣，定义知识边界
- 决策逻辑：
  - 话题在知识范围内 → 直接回答，不调工具
  - 话题在边缘 → 模糊回答（"好像是..."），可选择性调工具确认
  - 话题在范围外 → 调工具，但用"我搜了一下"包装
  - 话题完全超纲 → 直接说"我不太懂这个"
- 工具结果需要"消化"后融入回复，禁止原文搬运

#### 3.2.5 自发行为生成器

- 触发来源：时间引擎事件（长时间未对话）、记忆层（发现可分享内容）、外部事件（节日/天气变化）
- 生成候选行为 → 评估适当性（时段、关系、频率）→ 选择最自然的一个
- 频率约束：主动消息不超过 1 次/24h（防止骚扰感）

### 3.3 时间引擎（纵贯轴）

**职责边界：** 不生成内容，只控制"何时做、以什么节奏做"。纵贯所有层。

#### 3.3.1 四频率通道

| 频率 | 控制对象 | 数据来源 | 更新频率 |
|------|---------|---------|---------|
| 天-月级 | 关系阶段基调（热情/中性/克制） | 记忆层关系模型 | 每次对话结束评估 |
| 小时级 | 时段状态（活跃/忙碌/休息） | 系统时钟 + 人设日程模板 | 每小时 |
| 分钟级 | 情绪状态（兴奋/平静/低落） | 事件总线 emotional 事件 | 实时，带衰减函数 |
| 秒级 | 即时反应（立即回复/等待/中断） | 消息调度器分类结果 | 每条消息 |

#### 3.3.2 层叠仲裁协议

```
天-月级 设定 "默认基调"
  ↓
小时级 可覆盖 "节奏"（不改基调）
  ↓
分钟级 可覆盖 "节奏"（情绪影响，带衰减）
  ↓
秒级 "紧急中断"（仅限强情感信号触发，有冷却时间）
  ↓
→ 最终输出：回复时机 + 回复长度 + 回复基调
```

**仲裁规则：**

1. 低频设基调，高频调节奏——关系基调最稳定，即时反应最灵活
2. 紧急中断唯一触发条件：对方的强情感信号（悲伤、愤怒、危机）
3. 每个覆盖层有衰减——情绪兴奋的半衰期约 5-8 条消息，忙碌状态有时段边界

#### 3.3.3 情绪衰减函数

```
emotion_intensity(t) = initial_intensity × e^(-λt)

其中：
  initial_intensity = 事件触发时的情绪强度 (0-1)
  λ = 衰减速率（可按情绪类型配置，悲伤衰减慢 λ=0.05，兴奋衰减快 λ=0.15）
  t = 自触发以来的消息轮次
```

### 3.4 记忆与状态层

**职责边界：** 所有持久化数据的管理者。是自我状态的唯一写入者。

#### 3.4.1 三级记忆体系

| 层级 | 内容 | 存储方式 | 生命周期 |
|------|------|---------|---------|
| 即时记忆 | 当前 session 的完整对话 | 内存 | session 结束后摘要入工作记忆 |
| 工作记忆 | 近期 N 次对话的结构化摘要 | SQLite | 滚动窗口，自动淘汰最旧的 |
| 长期记忆 | 关键事件/承诺/共识/用户画像 | SQLite + 向量索引 | 永久，可被检索 |

#### 3.4.2 关系模型

```json
{
  "user_id": "xxx",
  "relationship_stage": "familiar",       // stranger → acquaintance → familiar → intimate → cooling
  "intimacy_score": 0.72,                 // 0-1, 由交互频率+深度+时长计算
  "interaction_count": 156,
  "last_interaction": "2026-04-03T22:30:00",
  "topics_shared": ["work", "movies", "cooking"],
  "emotional_history": [...],             // 重要情感事件记录
  "promises": [...],                      // bot 做过的承诺
  "user_profile": {                       // 渐进式构建的用户画像
    "inferred_age_range": "25-30",
    "communication_style": "casual",
    "sensitive_topics": ["family"]
  }
}
```

#### 3.4.3 自我状态

```json
{
  "current_mood_baseline": 0.6,           // 0=低落, 0.5=平静, 1=愉快
  "active_emotions": [
    {"type": "curiosity", "intensity": 0.3, "decay_rate": 0.1, "triggered_at": "..."}
  ],
  "recent_experiences": [                 // 可用于主动分享的素材
    {"type": "news_discovery", "content": "...", "timestamp": "..."}
  ],
  "current_persona_state": {
    "energy_level": "normal",             // 受时段影响
    "social_battery": 0.8,               // 连续对话后递减
    "identity_defense_log": [...]         // 最近的身份挑战记录
  }
}
```

### 3.5 基础设施层

#### 3.5.1 消息调度器

**入站调度：**

```
消息到达
  → 放入缓冲区，启动计时器 Δt
  → Δt 内有新消息到达 → 重置计时器
  → Δt 到期 → 消息分类器分类 → 打包上送认知层

Δt 动态参数：
  私聊正常时段: 2-4s
  私聊深夜时段: 8-15s
  群聊: 5-10s

分类标签：
  urgent_interrupt → 跳过缓冲（情绪危机、@提及、直接提问）
  append_type → 合并到前一条（"还有"、"对了"、纯表情补充）
  new_topic → 标记为新话题入口
```

**出站调度：**

```
认知层回复到达
  → 长度检查：>100字 → 拆分为多条
  → "正在输入..." 信号先发出
  → 按间隔逐条发送（3-8s/条，随机化）
  → 多话题回复按优先级排序发送
```

#### 3.5.2 LLM 调用管线

- Prompt 组装 pipeline: context assembly → persona injection → instruction → generation
- 调用合并策略：情绪推理 + 回复生成可合并为一次调用（通过 structured output）
- 降级策略：API 超时时返回预生成的"万能回复"（用人设语气的"稍等"类表达）
- 模型路由：轻量判断（分类/情绪检测）→ 小模型；核心生成 → 大模型

#### 3.5.3 工具运行时

| 工具 | 用途 | 调用方式 |
|------|------|---------|
| 浏览器引擎 | 实时信息检索、事实核查 | headless browser / search API |
| 知识库 | 人设相关领域知识 | 本地向量数据库 |
| 日历/天气 | 时间感知、话题素材 | API 调用 |
| 新闻/RSS | 主动分享素材来源 | 定时拉取 + 缓存 |

#### 3.5.4 本地持久化

- SQLite 作为主存储（记忆、关系、配置、事件日志）
- 向量索引（FAISS/ChromaDB）用于长期记忆检索
- 配置文件（YAML/JSON）热加载，支持不重启更新人设

#### 3.5.5 可观测性

- 事件总线日志自动归档（每条消息的完整处理链路）
- 关键指标看板：回复延迟分布、LLM 调用次数/成本、身份边界触发频率
- "为什么这样回复"的追溯能力：给定一条回复，可回溯其 prompt、检索到的记忆、时间引擎状态

### 3.6 事件总线（全局侧通道）

**设计原则：**

1. 各层只发射事件，不直接写共享状态
2. 记忆层是唯一的状态消费者/写入者
3. 事件是异步的，不阻塞请求流
4. 记忆层有合并逻辑，高频事件不会导致状态震荡

**核心事件类型：**

| 事件类型 | 发射者 | 状态影响 |
|---------|-------|---------|
| `conversation.emotional` | 认知层 | 更新情绪基线、情绪衰减参数 |
| `conversation.topic` | 认知层 | 更新话题栈、工作记忆 |
| `tool.discovery` | 基建层 | 更新"最近经历"（可分享素材） |
| `time.phase_shift` | 时间引擎 | 更新时段状态 |
| `time.relationship_eval` | 时间引擎 | 触发关系阶段重评估 |
| `identity.challenge` | 认知层 | 记录身份挑战事件及应对策略 |
| `proactive.trigger` | 时间引擎 | 激活自发行为生成器 |
| `system.degradation` | 基建层 | 触发降级模式 |

## 六、场景验证（端到端）

### 场景 1：深夜连发 + 情绪信号

> 用户深夜连发 3 条："在吗" → "睡了吗" → "我今天好累"

1. **消息调度器**：缓冲 Δt=10s，三条合并打包，标记"情绪信号"
2. **时间引擎仲裁**：亲密期(热情) → 深夜覆盖(+15s延迟) → 无紧急中断(疲惫≠危机) → 最终：热情但简短，延迟15s
3. **认知层**：检索记忆(用户近期加班) + 情绪推理(疲惫求安慰) → 共情回应，不展开
4. **感知层**：温柔语气 + 安慰表情包
5. **消息调度器出站**：拆分为 3 条短消息间隔发出
6. **事件总线**：`conversation.emotional` → 记忆层更新自我状态

### 场景 2：群聊 @问事实问题

1. **消息调度器**：@提及 → 跳过缓冲
2. **时间引擎**：工作时间，正常延迟
3. **认知层**：群聊策略(被@应回) → 工具决策(检查人设知识轮廓) → 调工具搜索 → 消化结果
4. **感知层**：群聊简洁风格
5. **身份守卫**：回复不含超人类特征 → 通过

### 场景 3：48h 未对话主动发起

1. **时间引擎**：`proactive.trigger` → 事件总线
2. **认知层自发行为生成器**：读记忆(上次聊面试) → 评估适当性(下午3点、关系中高) → 生成"面试准备得怎么样了？"
3. **时间引擎仲裁**：关系允许 + 时段合适 → 通过
4. **感知层**：轻松随意语气
5. **消息调度器出站**：直接发出
