## 一、问题全景：AI 语言指纹图谱

### 1.1 MVP 实测直接发现

| 编号 | 症状 | 示例 | 致命度 |
|------|------|------|--------|
| F01 | 多选项罗列 | "可以从以下角度思考：1)... 2)... 3)..." | 🔴 极高 |
| F02 | 疑问句收尾求确认 | "你觉得呢？""需要我展开吗？" | 🔴 极高 |
| F03 | 单条回复过长 | 一条 200+ 字，结构化段落 | 🔴 高 |
| F04 | 知识讲解体 | 自动进入教学模式，提供完整解释 | 🟠 高 |

### 1.2 深度分析：隐性 AI 特征

| 编号 | 症状 | 真人对照 | 致命度 |
|------|------|---------|--------|
| F05 | 永远积极/配合 | 真人会说"不想聊""没心情" | 🟠 高 |
| F06 | 过度共情模板化 | "我理解你的感受"每次都出现 | 🟠 中 |
| F07 | 零攻击性 | 真人会反驳、嘲讽、生气 | 🟠 高 |
| F08 | 万能开场白 | "好的""当然""没问题"起手 | 🟡 中 |
| F09 | 永远配合话题 | 真人会岔开话题、忽略部分问题 | 🟠 高 |
| F10 | 无信息不对称 | 真人会留悬念、不确定、模糊 | 🟡 中 |
| F11 | 零对话成本 | 真人需要暖场，不会直切话题 | 🟡 低 |
| F12 | 无记忆引用错误 | 真人会记混、张冠李戴 | 🟡 低 |

### 1.3 根因分析

```
                  ┌─────────────────────────────────────┐
                  │   LLM 的训练目标 = "有用的助手"        │
                  │   ↓ 产生天然倾向：                     │
                  │   · 尽可能全面回答（→ F01/F03/F04）    │
                  │   · 确认用户需求（→ F02）              │
                  │   · 保持礼貌积极（→ F05/F06/F07/F08） │
                  │   · 配合用户意图（→ F09/F10）          │
                  └─────────────────────────────────────┘
                              │
          ┌─────────────────────────────────────┐
          │ v4 现有防线只覆盖"表达层"（S5）       │
          │ 未覆盖"内容结构层"（S3/S4 prompt）   │
          │ 未覆盖"输出校验层"（S5 后置检查）     │
          └─────────────────────────────────────┘
```

**结论：** 需要三层防线，分别在 **Prompt 注入（S3/S4）**、**输出结构改写（S5 增强）**、**后置校验拦截（S5.5 新增）** 三个位置部署。

## 二、三层防御架构

```
Pipeline:
S1 → S2 → S3+S4 → S5 → [S5.5] → S6
              │        │       │
              │        │       └── 第三层：Anti-AI Validator（后置校验，拦截漏网）
              │        └────────── 第二层：Human-Style Rewriter（结构改写）
              └─────────────────── 第一层：Anti-AI Prompt Injection（源头约束）
```

### 架构原则

| 原则 | 说明 |
|------|------|
| **源头优先** | 70% 的 AI 味应在 Prompt 层消除，让 LLM 一开始就不生成问题内容 |
| **改写兜底** | S5 对 LLM 输出做结构级改写，不仅是表层"加口头禅" |
| **校验拦截** | S5.5 做最后一道检查，检测到 AI 指纹则触发局部重写或降级 |
| **可配置** | 所有规则强度从 persona.yaml 读取，不同人设可调整"去 AI 化"程度 |

## 三、第一层：Anti-AI Prompt Injection（S3/S4 Prompt 层）

### 3.1 Persona Schema 新增字段

```yaml
# persona.yaml 新增 anti_ai_speech 配置块
language:
  # ... 现有字段 ...
  anti_ai_speech:
    enabled: true
    strictness: 0.8              # 0-1，去 AI 化强度。0.8 = 严格

    # === 结构约束 ===
    max_message_length: 60        # 单条消息最大字数（中文字符）
    max_options_in_reply: 0       # 回复中最多列举选项数（0=禁止罗列选项）
    max_questions_per_reply: 1    # 单条回复最多包含问句数
    end_with_question_prob: 0.15  # 以问句结尾的概率（真人约 10-20%）

    # === 禁止模式 ===
    banned_patterns:
      - type: "enumeration"       # 编号列举: "1. 2. 3." 或 "首先...其次...最后"
        action: "rewrite"
      - type: "meta_question"     # 元问题: "需要我展开吗？""你还想了解什么？"
        action: "delete"
      - type: "hedge_opening"     # 万能开场: "好的""当然""没问题"
        action: "delete"
      - type: "empathy_template"  # 模板共情: "我理解你的感受""这确实不容易"
        action: "vary"            # 不删除而是变体化
      - type: "knowledge_dump"    # 知识倾倒: 主动解释对方没问的背景知识
        action: "truncate"

    # === 正向注入 ===
    human_behaviors:
      incomplete_answer: 0.3      # 概率：不完整回答（只答部分，剩下的之后再说）
      topic_drift: 0.1            # 概率：轻微跑题（联想到别的事）
      selective_ignore: 0.2       # 概率：忽略多问题中的部分（真人经常只回答自己想回答的）
      push_back: 0.15             # 概率：反驳或表达不同意见
      mood_refusal: 0.05          # 概率：以心情为由拒绝深度回答
```

### 3.2 Prompt 注入模板

在 S3+S4 的 System Prompt 中追加 **Anti-AI 行为指令块**，由 Parameter Interpreter 从上述配置生成：

```yaml
# prompt_mappings.yaml 新增
anti_ai_speech:
  strictness:
    [0.0, 0.3]: ""   # 低强度：不注入额外约束
    [0.3, 0.6]: |
      【回复风格约束】
      - 你在发消息，不是写文章。每条消息不超过{max_message_length}个字。
      - 不要列举选项让对方选，直接说你的想法。
      - 不要在结尾问"你觉得呢"或"需要我展开吗"。
    [0.6, 0.8]: |
      【回复风格·严格约束】
      - 你是在手机上打字聊天，不是在写邮件。
      - 每条消息最多{max_message_length}个字。想说多的就分几条发。
      - 禁止：列举（1. 2. 3.）、编号、"首先/其次/最后"结构。
      - 禁止：以"好的""当然""没问题"开头。
      - 禁止：以"你觉得呢？""需要展开吗？""还有什么想了解的？"结尾。
      - 禁止：主动解释对方没问的知识。别人没问就别讲。
      - 如果对方问了多个问题，你可以只回答其中你最想回答的1-2个。
      - 你有权表达"不太想聊这个"或"这个我也不确定啊"。
    [0.8, 1.0]: |
      【回复风格·最严格约束】
      - 你是一个真人在微信/飞书上和朋友聊天。
      - 每条消息不超过{max_message_length}字。长了就分条发。
      - 绝对禁止以下行为（一旦出现立即暴露你是 AI）：
        × 罗列选项 ("从以下几个方面..." "可以考虑...")
        × 结尾反问确认 ("你觉得呢" "要展开吗" "有帮助吗")
        × 万能开头 ("好的" "当然" "没问题" "确实")
        × 提供对方没要求的完整解释
        × 使用"首先...其次...最后"结构
        × 主动表示"如果你需要..."
      - 你应该：
        ✓ 只说你真正想说的，不是对方想听的
        ✓ 可以不回答不想回答的部分
        ✓ 可以说"不知道""不太想聊""改天再说"
        ✓ 可以只发一个"哈哈哈"或"嗯嗯"就完事
        ✓ 可以主动聊你自己感兴趣的事而不是围着对方转
```

### 3.3 Parameter Interpreter 扩展

```typescript
// parameter-interpreter.ts 新增
function resolveAntiAiSpeech(config: AntiAiSpeechConfig): string {
  if (!config.enabled) return '';

  const strictnessSegment = lookupMapping('anti_ai_speech.strictness', config.strictness);

  // 动态替换占位符
  return strictnessSegment
    .replace('{max_message_length}', String(config.max_message_length))
    .replace('{max_options_in_reply}', String(config.max_options_in_reply));
}
```

## 四、第二层：Human-Style Rewriter（S5 增强）

### 4.1 现有 S5 的不足

当前 S5 只做：
- 口头禅注入
- 错别字注入
- 消息拆分（按字数）
- emoji 注入

缺失：**对话结构层的改写**。即使 Prompt 约束了 LLM，仍有 20-30% 概率生成 AI 味内容（LLM 的训练惯性强于 prompt 约束）。S5 必须能检测并改写。

### 4.2 S5 新增：结构改写引擎

```typescript
interface AntiAiRewriteRule {
  id: string;
  detector: (text: string) => DetectionResult | null;
  rewriter: (text: string, detection: DetectionResult) => string;
  priority: number;  // 高优先级先执行
}

interface DetectionResult {
  matched: boolean;
  pattern_type: string;      // 匹配到的 AI 模式类型
  matched_segments: string[]; // 命中的文本片段
  confidence: number;        // 0-1
}
```

#### 规则清单

```typescript
const ANTI_AI_RULES: AntiAiRewriteRule[] = [

  // ═══ R01: 选项罗列消除 ═══
  {
    id: 'R01_enumeration_killer',
    detector: (text) => {
      // 检测模式：
      // 1) 数字编号: "1. xxx 2. xxx" "第一... 第二..."
      // 2) 顺序连接词: "首先...其次...最后" "一方面...另一方面"
      // 3) 冒号+换行列举: "xxx：\n- aaa\n- bbb"
      const patterns = [
        /[1-9][.、）)]\s*.+[1-9][.、）)]\s*.+/s,
        /首先.{5,}其次.{5,}/s,
        /一方面.{5,}另一方面.{5,}/s,
        /第[一二三四五].{3,}第[一二三四五].{3,}/s,
        /[:：]\s*\n\s*[-·•]\s*.+\n\s*[-·•]/s,
        /可以从以下.{0,5}(方面|角度|维度)/,
        /有(几个|以下|这些)(方面|原因|因素|建议)/,
      ];
      for (const p of patterns) {
        if (p.test(text)) {
          return { matched: true, pattern_type: 'enumeration',
                   matched_segments: [text.match(p)![0]], confidence: 0.9 };
        }
      }
      return null;
    },
    rewriter: (text, detection) => {
      // 改写策略：只保留列举中的第 1 项（或随机 1 项），
      // 其余删除。模拟真人"只说最重要的那个点"
      // 如果保留后仍超长，截断并加 "...算了说不清哈哈"
      return extractFirstPointOnly(text);
    },
    priority: 100,
  },

  // ═══ R02: 尾部问句清除 ═══
  {
    id: 'R02_tail_question_killer',
    detector: (text) => {
      const tailPatterns = [
        /[你您](觉得|认为|怎么看|看呢)[呢吗？?！!]*\s*$/,
        /(需要|要不要|想不想)(我)?(.{0,5})(展开|详细|解释|说说|聊聊)[吗？?]*\s*$/,
        /(还有|还想).{0,10}(了解|知道|聊|问)[的吗？?]*\s*$/,
        /(有帮助|有用|清楚)[吗了？?]*\s*$/,
        /对吧[？?]*\s*$/,
      ];
      for (const p of tailPatterns) {
        if (p.test(text)) {
          return { matched: true, pattern_type: 'meta_question',
                   matched_segments: [text.match(p)![0]], confidence: 0.95 };
        }
      }
      return null;
    },
    rewriter: (text, detection) => {
      // 直接删除尾部问句
      return text.replace(detection.matched_segments[0], '').trimEnd();
    },
    priority: 95,
  },

  // ═══ R03: 万能开场白清除 ═══
  {
    id: 'R03_hedge_opener_killer',
    detector: (text) => {
      const openerPatterns = [
        /^(好的[，,]?\s*)/,
        /^(当然[了啦，,]?\s*)/,
        /^(没问题[，,！!]?\s*)/,
        /^(确实[，,]?\s*)/,
        /^(嗯[，,]?\s*这个问题)/,
        /^(很好的问题[，,！!]?\s*)/,
        /^(这是一个很好的(问题|观点)[，,。.！!]?\s*)/,
        /^(我(很)?(理解|明白)[你您]的.{0,10}[，,。.]\s*)/,
      ];
      for (const p of openerPatterns) {
        if (p.test(text)) {
          return { matched: true, pattern_type: 'hedge_opening',
                   matched_segments: [text.match(p)![0]], confidence: 0.85 };
        }
      }
      return null;
    },
    rewriter: (text, detection) => {
      // 删除开场白，直接从实质内容开始
      return text.replace(detection.matched_segments[0], '').trimStart();
    },
    priority: 90,
  },

  // ═══ R04: 过长消息强制拆分+截断 ═══
  {
    id: 'R04_length_enforcer',
    detector: (text) => {
      // max_message_length 从 persona.yaml 读取
      if (text.length > persona.language.anti_ai_speech.max_message_length * 1.5) {
        return { matched: true, pattern_type: 'too_long',
                 matched_segments: [], confidence: 1.0 };
      }
      return null;
    },
    rewriter: (text) => {
      // 策略：
      // 1. 先按句号/感叹号/问号切分
      // 2. 取前 N 个句子使得总长 <= max_message_length
      // 3. 剩余部分丢弃（模拟真人"懒得打那么多字"）
      // 4. 概率性追加 "...算了打字太多了" 或直接截断
      return truncateToHumanLength(text);
    },
    priority: 80,
  },

  // ═══ R05: 知识讲解体压缩 ═══
  {
    id: 'R05_knowledge_dump_compressor',
    detector: (text) => {
      // 启发式：如果一段回复包含 > 3 个逗号分隔的长句，
      // 且不含情绪词/口语词 → 大概率是知识讲解体
      const sentences = text.split(/[，,。.！!？?]/);
      const longSentences = sentences.filter(s => s.trim().length > 15);
      const hasEmotionWord = /[哈嘿唉嗯啊呀吧呢嘛卧草靠牛逼绝了6️⃣]/.test(text);

      if (longSentences.length >= 3 && !hasEmotionWord) {
        return { matched: true, pattern_type: 'knowledge_dump',
                 matched_segments: [], confidence: 0.7 };
      }
      return null;
    },
    rewriter: (text) => {
      // 只保留核心观点（第一句或最后一句），
      // 用口语化表达替换，加入语气词
      return compressToOneLiner(text);
    },
    priority: 75,
  },

  // ═══ R06: 模板共情变体化 ═══
  {
    id: 'R06_empathy_variator',
    detector: (text) => {
      const templates = [
        /我(能)?(理解|明白)(你|您)的(感受|心情|处境|难处)/,
        /这(确实|真的)(很)?(不容易|辛苦|难|让人)/,
        /(抱歉|对不起).{0,5}(听到|知道)这(个|件事|些)/,
      ];
      for (const p of templates) {
        if (p.test(text)) {
          return { matched: true, pattern_type: 'empathy_template',
                   matched_segments: [text.match(p)![0]], confidence: 0.8 };
        }
      }
      return null;
    },
    rewriter: (text, detection) => {
      // 替换为更自然的共情表达（从人设词库随机）
      const naturalEmpathy = [
        '唉',
        '我靠',
        '卧槽',
        '...这也太',
        '天',
        '啊这',
        '不是吧',
      ];
      // 从 persona.vocabulary 中选择匹配人设的变体
      return replaceWithNaturalEmpathy(text, detection, naturalEmpathy);
    },
    priority: 70,
  },
];
```

### 4.3 S5 改写引擎执行流

```
S5 输入: CognitiveOutput (LLM 原始回复)
    │
    ├─ Step 1: Anti-AI 规则链（按 priority 降序执行）
    │   R01 选项罗列消除 → R02 尾部问句清除 → R03 开场白清除
    │   → R04 长度强制 → R05 知识讲解压缩 → R06 共情变体
    │
    ├─ Step 2: 现有 S5 流程（口头禅/错别字/emoji/消息拆分）
    │
    └─ Step 3: 输出 StyledMessages[]
```

**关键设计决策：Anti-AI 规则在现有 S5 流程之前执行。** 因为先改写结构再注入不完美，顺序更自然。

## 五、第三层：Anti-AI Validator（S5.5 新增 Stage）

### 5.1 为什么需要第三层

- 第一层（Prompt）覆盖率 ~70%，但 LLM 有训练惯性
- 第二层（S5 改写）基于正则和启发式，有漏网
- 需要一个**最终校验器**作为安全网

### 5.2 S5.5 Stage 定义

```
Stage: S5.5 · Anti-AI Validator
类型: Rule-based（零 LLM 成本）
位置: S5 之后，S6 之前
输入: StyledMessages[]
输出: ValidatedMessages[] | RewriteRequest

逻辑:
  1. 对每条 StyledMessage 运行 AI 指纹评分器
  2. 计算 ai_score (0-100)
  3. 判定：
     - ai_score < 30 → PASS，进入 S6
     - 30 ≤ ai_score < 60 → WARN，发射 event，但仍通过
     - ai_score ≥ 60 → BLOCK，触发降级回复或请求 S5 重写
```

### 5.3 AI 指纹评分器

```typescript
interface AiFingerprint {
  feature: string;
  weight: number;
  detector: (messages: StyledMessage[]) => number; // 返回 0-1 的命中强度
}

const AI_FINGERPRINTS: AiFingerprint[] = [
  {
    feature: 'enumeration',
    weight: 20,
    detector: (msgs) => {
      // 检测是否还存在编号、列举结构
      const fullText = msgs.map(m => m.text).join(' ');
      return /[1-9][.、]\s|首先.+其次|一方面.+另一方面/.test(fullText) ? 1.0 : 0;
    }
  },
  {
    feature: 'tail_meta_question',
    weight: 20,
    detector: (msgs) => {
      const lastMsg = msgs[msgs.length - 1].text;
      return /你(觉得|怎么看)|需要.*(展开|详细)|有(帮助|用)吗|对吧[？?]/.test(lastMsg) ? 1.0 : 0;
    }
  },
  {
    feature: 'excessive_length',
    weight: 15,
    detector: (msgs) => {
      const maxLen = persona.language.anti_ai_speech.max_message_length;
      const overLengthCount = msgs.filter(m => m.text.length > maxLen).length;
      return Math.min(overLengthCount / msgs.length, 1.0);
    }
  },
  {
    feature: 'hedge_opening',
    weight: 10,
    detector: (msgs) => {
      const first = msgs[0].text;
      return /^(好的|当然|没问题|确实|我理解)/.test(first) ? 1.0 : 0;
    }
  },
  {
    feature: 'zero_emotion_words',
    weight: 10,
    detector: (msgs) => {
      // 如果整段回复不含任何口语/情绪词 → 偏 AI
      const fullText = msgs.map(m => m.text).join('');
      const emotionWords = /[哈嘿唉嗯啊呀吧呢嘛靠卧草天呐emmm]|\.{3}|！{2,}|哈{2,}/;
      return emotionWords.test(fullText) ? 0 : 0.7;
    }
  },
  {
    feature: 'perfect_structure',
    weight: 10,
    detector: (msgs) => {
      // 如果每条消息都是完整句子（有主谓宾、正确标点）→ 偏 AI
      const allComplete = msgs.every(m => /[。.！!？?]$/.test(m.text.trim()));
      return allComplete ? 0.6 : 0;
    }
  },
  {
    feature: 'no_self_reference',
    weight: 5,
    detector: (msgs) => {
      // 如果回复中没有"我"字（全在讲道理/解释）→ 偏 AI
      const fullText = msgs.map(m => m.text).join('');
      return /我/.test(fullText) ? 0 : 0.5;
    }
  },
  {
    feature: 'knowledge_density',
    weight: 10,
    detector: (msgs) => {
      // 信息密度过高 → 偏 AI（真人聊天信息密度低）
      const fullText = msgs.map(m => m.text).join('');
      const totalChars = fullText.length;
      // 统计"知识性"标记：数字、英文术语、括号解释
      const knowledgeMarkers = (fullText.match(/[\d]+|[a-zA-Z]{3,}|[（(][^)）]+[)）]/g) || []).length;
      const density = knowledgeMarkers / Math.max(totalChars / 20, 1);
      return Math.min(density, 1.0);
    }
  },
];

function calculateAiScore(messages: StyledMessage[]): number {
  let score = 0;
  for (const fp of AI_FINGERPRINTS) {
    const hitStrength = fp.detector(messages);
    score += fp.weight * hitStrength;
  }
  return Math.min(Math.round(score), 100);
}
```

### 5.4 BLOCK 时的降级策略

当 ai_score ≥ 60 时，S5.5 不直接返回该回复，而是：

```
BLOCK 路径:
  1. 尝试一次：回退到 S5，强制开启所有 Anti-AI 规则的最高强度重写
  2. 重写后再次评分
  3. 如果仍 ≥ 60 → 使用预生成的"人类模板回复"替代：
     - 对应 response_strategy 的轻量回复模板
     - 例如 strategy=casual → 随机选 ["嗯嗯", "哈哈好", "确实", "是哦"]
     - 例如 strategy=empathize → 随机选 ["唉...", "抱抱", "别想太多啦"]
  4. 发射 event: anti_ai.block_triggered → 记录到事件日志供后续分析
```

## 六、Pipeline 更新后的完整流程

### 6.1 更新后的 Pipeline 图

```
S1 → S2 → S3+S4 → S5(enhanced) → S5.5 → S6
              │          │            │
              │          │            └── ai_score < 30: PASS
              │          │                ai_score 30-60: WARN + PASS
              │          │                ai_score ≥ 60: BLOCK → 重写/降级
              │          │
              │          ├── Anti-AI 规则链（R01-R06）
              │          └── 现有流程（口头禅/错别字/拆分）
              │
              └── System Prompt 含 Anti-AI 行为约束指令
```

### 6.2 性能影响评估

| 层级 | 类型 | 额外延迟 | LLM 成本 |
|------|------|---------|---------|
| 第一层 (Prompt) | System Prompt 加长 ~200 token | +0ms（与 S3/S4 合并） | +约 200 input tokens/次 |
| 第二层 (S5 改写) | 正则+字符串操作 | +5-15ms | 0 |
| 第三层 (S5.5 校验) | 正则+评分 | +5-10ms | 0 |
| **总计** | — | **+10-25ms** | **+~200 tokens** |

**结论：** 近乎零成本。所有 Anti-AI 层都是 rule-based，不增加 LLM 调用。
