## 一、问题全景

### 1.1 现象

MVP 测试对话：
```
用户: 你什么时候开始了解 MBTI 的？
Bot:  高三那会儿吧，备考压力大，同学推荐我测了一次...
      [7 轮对话后，窗口外]
用户: 你之前说高三测过 MBTI？
Bot:  我大一才第一次测的，心理学课上老师让我们做的...
```

用户感知：**在说谎。** 信任瞬间崩塌。

### 1.2 根因递进链

```
问题1: 即兴生成无约束 → LLM 自行推演传记，每次可能不同
    ↓
问题2: 无返写机制 → 说过就忘，窗口外不可追溯
    ↓
问题3: 返写无控制 → 人生轨迹膨胀，事实间互相矛盾
    ↓
问题4: 无一致性校验 → 新生成内容无法与已有记录对账
```

这四个问题是**递进依赖关系**，解决了前一个才会遇到后一个。

### 1.3 现有系统缺失

| 组件 | 现状 | 问题 |
|------|------|------|
| persona.yaml | 只有 `age: 19` + 基本背景 | 无人生经历数据，LLM 只能瞎编 |
| important_events 表 | 存在但 0 条记录 | 从未被写入 |
| S2 上下文组装 | 检索 working_memory + long_term_memory | 不检索传记事实 |
| S3+S4 Prompt | 无传记约束 | LLM 不知道"以前说过什么" |
| 滑动窗口 | 25-50 轮 immediate_memory | 窗口外的即兴内容完全丢失 |

## 二、解决方案：三层传记记忆 + 一致性门控

### 2.0 架构总览

## 三、第一层：传记锚点（Biographical Anchors）

### 3.1 Persona Schema 新增

```yaml
# persona.yaml 新增 biography 配置块
biography:
  # === 人生时间线锚点（作者预定义）===
  anchors:
    - period: "童年"
      age_range: [0, 6]
      location: "深圳"
      facts:
        - "在深圳出生长大"

    - period: "小学"
      age_range: [6, 12]
      location: "深圳"
      facts:
        - "开始学游泳"
        - "四年级转学过一次"

    - period: "初中"
      age_range: [12, 15]
      location: "深圳"
      facts:
        - "开始听独立音乐"

    - period: "高中"
      age_range: [15, 18]
      location: "香港"
      facts:
        - "高一转学到香港"
        - "开始接触冲浪"
        - "高三备考压力大，第一次测 MBTI"

    - period: "大学"
      age_range: [18, 19]
      location: "香港"
      facts:
        - "港大心理学大一"
        - "开始玩胶片摄影"
        - "认识了一个冲浪搭子"

  # === 空白期声明（作者显式标注没有设定的时段）===
  blank_periods:
    - period: "初中"
      note: "这段时间没什么特别的，可以自由发挥但不能编造重大事件"

  # === 绝对禁止编造的内容 ===
  forbidden_fabrications:
    - "重大疾病或住院经历"
    - "出国留学（除香港外）"
    - "恋爱细节"
    - "家庭成员具体信息"

  # === 返写控制 ===
  writeback:
    enabled: true
    max_facts_per_period: 5       # 每个时间段最多生成事实数
    max_total_facts: 50           # 生成事实总上限
    importance_threshold: 0.3     # 低于此重要度不写入
    conflict_strategy: "reject"   # 冲突时策略：reject（拒绝写入）| overwrite | ask
```

### 3.2 Schema 设计要点

**为什么需要 `blank_periods`：** 如果没有显式标注"这段时间可以自由发挥"，LLM 会因为缺乏信息而要么编造重大事件，要么完全回避该时段。blank_periods 给 LLM 一个明确信号："这段时间平淡无奇，你可以编一些小事但别编大事。"

**为什么需要 `forbidden_fabrications`：** 有些话题即使 LLM 编得很合理也不应该碰——涉及隐私、敏感、或一旦编错难以修正的领域。这是一条**硬围栏**。

### 3.3 Prompt 注入格式

传记锚点在 S3/S4 的 System Prompt 中以如下格式注入：

```
【你的人生经历】
以下是你真实经历过的事，回复时必须与这些一致：
- 小学(6-12岁)在深圳：开始学游泳，四年级转学过一次
- 高中(15-18岁)在香港：高一转学到香港，开始接触冲浪，高三第一次测MBTI
- 大学(18-19岁)在香港：港大心理学大一，开始玩胶片摄影

以下是你以前聊天时提到过的事（必须保持一致）：
- [高中] 高三班主任是个很严格的女老师（2024-04-01 对话中提到）
- [大学] 胶片相机是二手买的富士的（2024-04-02 对话中提到）

绝对不要编造：重大疾病、出国留学（除香港外）、恋爱细节、家庭成员具体信息。
如果被问到你不确定的过去经历，用模糊表达："好像是...""我记不太清了"。
```

## 四、第二层：生成事实返写（Generated Facts Writeback）

### 4.1 提取时机：S4.5 Fact Extractor（新增 Stage）

```
Pipeline 位置: S3+S4 之后，S5 之前
             S1 → S2 → S3+S4 → [S4.5] → S5 → S5.5 → S6
                                  │
                                  └── 从 LLM 原始输出中提取传记事实
                                      写入前做冲突检测
                                      异步写入，不阻塞 Pipeline

类型: LLM-light（Haiku 轻量调用）或 rule-based
```

**为什么不在 S5：** S5 是感知包装层，此时文本已经被注入错别字、口头禅、消息拆分。从改写后的文本中提取语义事实，准确率会大幅下降。必须在 LLM 原始输出（S3+S4 output）上提取。

**为什么是异步：** 事实提取和写入不影响用户等待时间。S4.5 将提取任务发射到事件总线，由记忆层异步消费。

### 4.2 提取逻辑

```typescript
interface BiographicalFact {
  id?: number;
  user_id: string;
  period: string;                    // 匹配 anchors 中的 period 或推断
  age_approx?: number;               // 推断的大致年龄
  fact_content: string;              // 具体事实
  source_type: 'anchor' | 'generated';  // 来源：预定义 vs 即兴生成
  source_conversation_id: string;    // 生成该事实的对话 session
  source_message_timestamp: number;  // 生成时的时间戳
  importance: number;                // 0-1
  confidence: number;                // 提取置信度 0-1
  is_active: boolean;               // false = 已被淘汰或标记冲突
  created_at: number;
}

// === 提取触发条件 ===
function shouldExtractFacts(cognitiveOutput: CognitiveOutput): boolean {
  const text = cognitiveOutput.reply_content;

  // 条件 1：包含时间标记
  const timeMarkers = /小时候|小学|初中|高中|大学|大一|去年|前年|以前|那会儿|岁的时候|记得有一次|刚来.*的时候/;

  // 条件 2：包含具体事件（动词 + 具体名词）
  const eventMarkers = /开始.{1,10}|第一次.{1,10}|学会了.{1,10}|去了.{1,10}|认识了.{1,10}|买了.{1,10}/;

  return timeMarkers.test(text) && eventMarkers.test(text);
}

// === 提取方式 ===
// 方式 A：轻量 LLM（Haiku，推荐，准确率高）
const EXTRACT_PROMPT = `
从以下回复中提取传记事实。只提取具体的人生经历，不提取观点或情感。
返回 JSON 数组，每项包含：period（时间段）、fact（事实）、importance（0-1）。
如果没有传记事实，返回空数组 []。

回复内容：
{reply_content}

已知时间段：童年、小学、初中、高中、大学
`;

// 方式 B：规则引擎（零 LLM 成本，准确率较低）
function ruleBasedExtract(text: string): BiographicalFact[] {
  const facts: BiographicalFact[] = [];
  // 正则匹配 "时间标记 + 事件" 模式
  const pattern = /(小学|初中|高中|大学|大一|小时候|那时候|那会儿).{0,5}(，|,|\s)(.{5,40})/g;
  let match;
  while ((match = pattern.exec(text)) !== null) {
    facts.push({
      period: normalizePeriod(match[1]),
      fact_content: match[3].trim(),
      source_type: 'generated',
      importance: 0.5,  // 默认中等，后续可调
      confidence: 0.6,
    });
  }
  return facts;
}
```

### 4.3 写入前冲突检测（盲区补充 #2）

**场景：** 第 10 次对话 bot 说"高三开始冲浪"，第 30 次说"大学才学的冲浪"。

```typescript
interface ConflictCheckResult {
  has_conflict: boolean;
  conflicting_fact?: BiographicalFact;
  conflict_type: 'time_mismatch' | 'detail_contradiction' | 'none';
  resolution: 'reject_new' | 'keep_both_flagged' | 'merge';
}

async function checkConflictBeforeWrite(
  newFact: BiographicalFact,
  existingFacts: BiographicalFact[]
): Promise<ConflictCheckResult> {

  // === 规则 1：同一主题不同时段 ===
  // 提取关键词，检查是否已有同主题但不同 period 的事实
  const keywords = extractKeywords(newFact.fact_content); // e.g. ["冲浪", "学"]
  const sameTopicFacts = existingFacts.filter(f =>
    f.is_active && keywords.some(kw => f.fact_content.includes(kw))
  );

  for (const existing of sameTopicFacts) {
    if (existing.period !== newFact.period) {
      return {
        has_conflict: true,
        conflicting_fact: existing,
        conflict_type: 'time_mismatch',
        resolution: 'reject_new',  // 默认保留先生成的（先入为主）
      };
    }
  }

  // === 规则 2：同时段矛盾细节 ===
  const samePeriodFacts = existingFacts.filter(f =>
    f.is_active && f.period === newFact.period
  );
  // 语义相似但细节不同（需轻量 LLM 判断，或简单字符串相似度）
  for (const existing of samePeriodFacts) {
    const similarity = stringSimilarity(existing.fact_content, newFact.fact_content);
    if (similarity > 0.5 && similarity < 0.9) {
      // 相似但不相同 → 可能矛盾
      return {
        has_conflict: true,
        conflicting_fact: existing,
        conflict_type: 'detail_contradiction',
        resolution: 'reject_new',
      };
    }
    if (similarity >= 0.9) {
      // 几乎相同 → 重复，合并
      return {
        has_conflict: false,
        conflict_type: 'none',
        resolution: 'merge',
      };
    }
  }

  // === 规则 3：与传记锚点矛盾 ===
  const anchors = getAnchorsForPeriod(newFact.period);
  for (const anchor of anchors) {
    if (contradicts(anchor, newFact.fact_content)) {
      return {
        has_conflict: true,
        conflict_type: 'detail_contradiction',
        resolution: 'reject_new',  // 锚点优先级绝对高于生成事实
      };
    }
  }

  return { has_conflict: false, conflict_type: 'none', resolution: 'merge' };
}
```

### 4.4 淘汰机制

```
写入前检查：
  1. 该 period 已有 >= max_facts_per_period 条？
     → 淘汰该 period 中 importance 最低 + retrieval_count 最少的一条
  2. 总量已有 >= max_total_facts 条？
     → 淘汰全局中 importance × recency_score 最低的一条
  3. 新事实 importance < importance_threshold？
     → 不写入（"高三吃了个三明治"级别的琐事）

recency_score = 1 / (1 + days_since_creation / 30)
```

## 五、第三层：一致性门控（Consistency Gate）

### 5.1 S2 检索注入

```typescript
// s2-context-assembler.ts 新增
async function retrieveRelevantBiography(
  messageContent: string,
  persona: PersonaConfig
): Promise<string> {

  // Step 1: 从消息中提取话题关键词
  const keywords = extractTopicKeywords(messageContent);
  // e.g. "你什么时候学的冲浪" → ["冲浪", "学"]

  // Step 2: 检索匹配的传记事实（anchors + generated）
  const relevantFacts = await biographyDAO.searchByKeywords(
    keywords,
    { limit: 8, only_active: true }
  );

  // Step 3: 按 period 排序，格式化为 Prompt 片段
  if (relevantFacts.length === 0) return '';

  const factsByPeriod = groupByPeriod(relevantFacts);
  let prompt = '【你的相关经历（回复时必须与以下一致）】\n';

  for (const [period, facts] of Object.entries(factsByPeriod)) {
    const factList = facts.map(f => {
      const sourceTag = f.source_type === 'anchor' ? '' : `（你在之前聊天中提到的）`;
      return `- ${f.fact_content}${sourceTag}`;
    }).join('\n');
    prompt += `${period}：\n${factList}\n`;
  }

  // Step 4: 注入 forbidden_fabrications
  if (persona.biography?.forbidden_fabrications?.length) {
    prompt += '\n绝对不要编造：' + persona.biography.forbidden_fabrications.join('、') + '。\n';
  }

  // Step 5: 注入模糊化指令
  prompt += '如果被问到你不确定的过去经历，用模糊表达：比如"好像是...""我记不太清了""大概是那时候？"。\n';

  return prompt;
}
```

### 5.2 注入量控制

| 参数 | 值 | 说明 |
|------|-----|------|
| 每次注入上限 | 8 条事实 | 避免 Prompt 膨胀 |
| 优先级排序 | confidence DESC, importance DESC, retrieval_count ASC | 高置信高重要度优先，常被检索的说明已被用户多次问及 |
| Token 预算 | ~200 tokens | 8 条 × ~25 字 = ~200 字 ≈ 200 tokens |
| anchor 优先 | 是 | anchor 类型事实排序始终高于 generated |

### 5.3 写入后一致性校验（可选 Phase 2）

在 S4.5 提取事实后，可选地用轻量 LLM 校验本次回复与已有事实是否矛盾：

```typescript
// 仅在 persona.biography.writeback.conflict_strategy !== 'reject' 时使用
const CONSISTENCY_CHECK_PROMPT = `
判断以下回复是否与已知事实矛盾。

已知事实：
{existing_facts}

本次回复：
{reply_content}

如果矛盾，返回 {"consistent": false, "conflict": "矛盾描述"}。
如果一致，返回 {"consistent": true}。
`;
```

**成本评估：** 使用 Haiku，~100 input tokens + ~20 output tokens。仅在检测到传记内容时触发，不是每条消息都跑。预计 10-20% 的对话会触发。

## 六、容错层：记忆模糊化（Memory Blur）

### 6.1 Persona Schema 新增

```yaml
# persona.yaml → language.imperfection 新增
imperfection:
  typo_rate: 0.03
  correction_behavior: true
  incomplete_thought_rate: 0.05
  filler_words: ["emmm", "这个嘛", "怎么说呢"]
  # ↓ 新增
  memory_blur:
    enabled: true
    blur_rate: 0.15                 # 15% 概率对回忆使用模糊表达
    blur_expressions:
      - "好像是..."
      - "我记得大概是..."
      - "应该是...吧？"
      - "emmm 具体不太记得了"
      - "大概那个时候？"
    blur_triggers:                  # 触发模糊化的话题类型
      - "specific_date"             # 具体日期
      - "exact_sequence"            # 精确顺序
      - "low_importance_detail"     # 低重要度细节
```

### 6.2 模糊化执行逻辑

```typescript
// 在 S5 或 S4.5 中执行
function applyMemoryBlur(
  replyContent: string,
  relatedFacts: BiographicalFact[],
  config: MemoryBlurConfig
): string {
  if (!config.enabled) return replyContent;
  if (Math.random() > config.blur_rate) return replyContent;

  // 只对 generated 类型的事实引用做模糊化（anchor 是确定的，不模糊）
  const generatedRefs = relatedFacts.filter(f => f.source_type === 'generated');
  if (generatedRefs.length === 0) return replyContent;

  // 在回复开头或事实引用前插入模糊表达
  const blurExpr = randomChoice(config.blur_expressions);

  // 策略：在时间标记前插入模糊化前缀
  return replyContent.replace(
    /(小学|初中|高中|大学|大一|那时候|那会儿)/,
    `${blurExpr}$1`
  );
}
```

### 6.3 为什么这很重要

模糊化不是"偷懒"，是**真人记忆的核心特征**：
- 真人回忆时天然带不确定性："好像是高三？还是高二？"
- 模糊化让偶尔的不一致变成"记忆不太清楚"而不是"在说谎"
- 这是一个**兜底安全网**——即使一致性门控漏了矛盾，模糊化也能让用户觉得合理

## 七、Cron Job 联动（盲区补充 #3）

### 7.1 Memory Maintenance Cron Job 扩展

在 v4.1 已有的 Cron Job 中增加传记事实维护任务：

| 频率 | 任务 | 内容 |
|------|------|------|
| **每周** | 传记事实聚类 | 合并语义相似的 generated 事实（"高三开始冲浪" + "高三的时候学了冲浪" → 保留一条） |
| **每周** | 时间线矛盾检测 | 扫描所有 active 事实，检测同一主题出现在不同 period 的矛盾 → 标记 `is_active=0` + 记录 `conflict_with_id` |
| **每月** | 传记密度审计 | 统计各 period 的事实数量分布，如果某个 period 远超其他 → 告警"该时段过于丰富，可能需要作者补充 anchors" |
| **每月** | 传记一致性评分 | 抽样 10 条 generated 事实 vs anchors → 计算一致性分数 |

### 7.2 矛盾检测算法

```typescript
async function detectTimelineConflicts(): Promise<ConflictReport[]> {
  const allFacts = await biographyDAO.getAllActive();
  const conflicts: ConflictReport[] = [];

  // 按关键词分组
  const factsByTopic = groupByTopicKeywords(allFacts);

  for (const [topic, facts] of Object.entries(factsByTopic)) {
    const periods = new Set(facts.map(f => f.period));
    if (periods.size > 1) {
      // 同一主题出现在多个时段 → 潜在矛盾
      conflicts.push({
        topic,
        facts,
        periods: [...periods],
        severity: facts.some(f => f.source_type === 'anchor') ? 'critical' : 'warning',
        suggested_action: 'deactivate_newer',  // 保留更早生成的
      });
    }
  }

  return conflicts;
}
```

### 8.2 S4.5 Stage 定义

```
Stage: S4.5 · Biographical Fact Extractor
类型: LLM-light (Haiku) 或 rule-based
位置: S3+S4 之后，S5 之前
执行方式: 异步（不阻塞主 Pipeline）

输入: CognitiveOutput (LLM 原始回复)
输出: 无直接输出（通过事件总线异步写入）

逻辑:
  1. shouldExtractFacts(cognitiveOutput) → 判断是否包含传记内容
  2. 如果是 → 调用 extractFacts() 提取 <period, fact, importance>
  3. 对每条提取的事实 → checkConflictBeforeWrite()
  4. 无冲突 → 检查容量上限 → 写入 biographical_facts 表
  5. 有冲突 → 按 conflict_strategy 处理（默认 reject）
  6. 发射事件: biography.fact_extracted 或 biography.conflict_detected
```

## 九、数据膨胀控制总结

```
固定层（anchors）         → 作者控制，永不膨胀
                            典型数据量：5-8 个时段 × 2-4 条/段 = 10-32 条
                            存储：~640 字

生成层（writeback）       → 上限 50 条，按重要性淘汰
                            存储：50 × 20 字 = ~1000 字
                            写入前冲突检测，Cron Job 周期清理

注入量（per request）     → 最多 8 条相关事实
                            Token 成本：~200 tokens/次

总内存占用               → ~2KB
总 DB 存储               → ~10KB（含索引）
额外 LLM 成本            → Haiku 调用，仅 10-20% 对话触发
                            ~120 tokens/次 × 0.15 = ~18 tokens 均摊/条消息
```
